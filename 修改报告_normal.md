# 修改报告 Github提示词版

## 修改1
1. 评估结果
平均分：55.50
见 evaluation_normal_1.json
2. 问题分析
a. 格式认知灾难与全局污染大爆发（致命错误）
由于代码中引入了 REQ_FORMAT_CONTENT（包含了【需求价值】、【需求场景】等标签），大模型误以为输出的每个子需求也必须包含这些结构。导致在 Row 2, 5, 19, 32, 39 中，大模型疯狂地在 description 里重复抄写这些全局信息，造成了极为严重的全局污染。
b. JSON 骨架崩塌（Row 17 得 0 分的原因）
同样受上述问题影响，大模型把精力全放在了写“需求场景”上，竟然在 Row 17 和 Row 19 中直接把 acceptance_criteria 这个必填的数组字段给弄丢了（直接把 AC 写进了 description 里），导致格式校验彻底失败，裁判直接给出了 0 分的重罚。
c. 对 CRUD（增删改查）拆解的“顽固免疫”
大模型对“动词”的理解依然存在死角。在 Row 39（新增、编辑、删除成员国配置）和 Row 32（固定数据的增删改）中，大模型认为“既然操作的都是同一个东西（成员国配置），那就放一个需求里吧”。它忽略了我们在 V7 强调的 CRUD 强制拆分原则。
d. 协议与依赖错位（局部/全局混淆）
在 Row 8 中，“DATEX/ASN 协议”、“TCP/IP”、“ESRI 产品”被塞进了子需求；Row 14 中的“VxWorks 5.3 操作系统”也被塞进子需求。大模型未能正确地将这些宏观的 IT 架构约束提取到 global_info 中。
3. 修改建议
a. 强力纠正输入输出认知（Input vs Output）
必须在提示词中明确大喝一声：“REQ_FORMAT_CONTENT 是提供给你的阅读材料格式，绝对不是你要输出的格式！”严禁在子需求中重复输出【需求价值】等标题。
b. 重申 JSON Schema 铁律
在指令的最醒目位置，强制要求必须包含 acceptance_criteria 数组，且格式必须是列表。
c. 升级 CRUD 拆解规则（锁定宾语陷阱）
专门针对“同一个对象”的增删改查下达禁令：“即使操作的宾语完全相同（例如：新增配置、修改配置、删除配置），只要是不同的动作，就必须拆散！”
d. 强化架构级约束提取
在 global_info 的提取规则中，直接点名“TCP/IP协议、底层操作系统、依赖的第三方软件产品”必须捞入全局。

## 修改2
1. 评估结果
平均分：84.00
见 evaluation_normal_2.json 
2. 问题分析
a. “修饰语/限制条件”被过滤（微观实体丢失）
大模型现在能准确抓取核心动作（动词），但经常把定语、状语和补充要求给丢了。
比如 Row 2 抓到了“打开应用”，但丢了“目录名需与应用名一致”；
Row 19 抓到了“单比特错误校正”，但丢了“≤100微秒”；
Row 25 抓到了“类接口”，但丢了具体的类名 SceMi。
b. 对“不同性质的动作”缺乏敏感度（动词重叠残留）
大模型依然偶尔会将不同阶段、不同性质的词合并：
查询与控制：Row 8 将“查询 DMS 状态”和“控制 DMS 信标”合并了（读写未分离）。
流程动作：Row 32 将“调整灵敏度”和“选择公差”合并了。
采集与输出：Row 14 将“采集光电二极管”与“输出数据包”合并了。
c. 边缘全局变量遗漏
在 Row 14 中，原文提到的 ROM&RAM 这种非功能性限制被大模型完全忽略了，因为它不知道该把这个放在哪里（我们的 Schema 没有明确点名）。
d. 裁判与大模型的“全局/局部”认知冲突
在 Row 14 中，大模型把“VxWorks 5.3”放在了全局，但裁判竟然报错说“这应该放进子需求”。这说明对于某些特定组件的依赖，裁判认为属于局部限制。我们需要在提示词里给出更灵活的“漏斗标准”。
3. 修改建议
a. 强化“定语/限制”的像素级透传：在规则中明文规定，抓取动作时必须携带所有修饰语（如类名、时间要求、目录一致性等具体条件），严禁将其作为废话省略。
b. 增加“读写分离”和“输入输出分离”规则：明确规定“查询/读取”与“控制/修改”必须拆开；“采集”与“生成/输出”必须拆开。
c. 扩充 JSON Schema 的全局提取范围：在 technical_and_env_constraints 的描述中，显式加入 ROM&RAM、硬件存储。
d. 细化全局/局部界限：告诉大模型：“如果是整个系统通用的 OS 放全局；如果是特定硬件/模块单独依赖的 OS 或协议，必须放在具体的子需求中！”

## 修改3
1. 评估结果
平均分：87.50（包含3个满分）
见 evaluation_normal_3.json
2. 问题分析
a. “状态切换”与“CRUD”的拆分界限冲突（过度拆分的反噬）
大模型现在对“拆分”过于敏感，导致了语义上的误判：
Row 19 中，大模型把“启用压缩”和“禁用压缩”拆成了两个需求。但裁判指出，启用/禁用属于同一个功能的正反状态，应该合并！
而在 Row 2 中，遇到“增删改查（CRUD）”，大模型却又把“删除”给漏了。这说明大模型没有建立“生命周期不同（CRUD）”与“功能开关（Enable/Disable）”的明确区分标准。
b. 准备动作与核心动作未分离
在 Row 14 中，大模型把“切换模式”和“采集图像”合在了一起。但在测试规范中，【模式切换（前置准备）】和【数据采集（核心执行）】是两个独立的测试用例，必须拆开。
c. 微观修饰语、量词与列表长尾的“磨损”
大模型在抓取实体时，经常会丢掉定语和状语：
Row 17 漏掉了具体的量词（1帧、3-6帧）。
Row 19 抓到了“记录”和“复位”，但漏掉了“二进制位记录”和“处理器复位”。
Row 30 在一长串字段名中，漏掉了 Input Identifier。
d. 代码级标志位 vs 行业通用协议的“全局/局部”倒置
大模型把特定的系统变量（IS_SETTLED=false）当成了全局条件（Row 17）。
却又把行业通用的标准和系统格式（MS-DOS格式、TMDD标准）当成了局部条件，在各个子需求里重复写（Row 8, Row 19）。
3. 修改建议
a. 建立“合并与拆分”的绝对分水岭：在提示词中明确规定：“启用/禁用”必须合并写在同一个需求的两个 AC 里；而“增、删、改”必须坚决拆散。
b. 斩断“准备->执行”链条：增加特殊指令，遇到“进入/切换某模式，并开始采集”时，强制切为两个需求。
c. 强化“量词与前置定语”的提取免疫：要求模型在提取名词时，必须“连带前方的形容词、量词、状态约束”一起抄录，绝不能只抓主干词。
d. 为 global_info 加上白名单/黑名单：明确行业标准（TMDD）、文件系统（MS-DOS）进全局，且子需求不可重复；明确“代码级变量（IS_XXX）”死也不能进全局。

## 修改4
1. 评估结果
平均分：87.00 （包含 2 个满分）
见 evaluation_normal_4.json
2. 问题分析
a. 提示词示例泄漏（典型幻觉）
在 Row 39 中，原文根本没有提到 TMDD 标准和 MS-DOS 文件系统，但大模型硬生生把它们写进了 global_info。这是因为我们在上一版 V10 的提示词里，为了教它分类，举了 (如TCP,TMDD)、(如MS-DOS) 的例子。大模型“生搬硬套”，把例子当成了真实数据捏造了出来。
b. 并列动作与业务流依然“藕断丝连”（动词重叠）
大模型对“增删改查”敏感了，但对非标准动作依然合并：
平行并列动作：Row 8 把“锁定摄像头、设置方向、预设位置”三个完全平行的按钮级动作合并了；Row 5 把“查看、选择、导航、搜索”合并了。
先后业务流：Row 32 把“用户登录 -> 系统校验 -> 权限分配”整个链路打成了一个包。
c. 全局 vs 局部的“系统级/模块级”判定失误
Row 5 中，大模型把“SDT需以有向无环图呈现”放进了全局。但裁判指出，这只是 SDT这个单独模块 的规则，不能算全局（系统级）规则。
Row 8 中，大模型把“设备控制需校验用户名密码”放在了局部。但裁判指出，这是所有设备控制的统一安全规则，应该放全局。大模型缺乏“是否影响全系统”的判断力。
d. 尾部长难句细节丢失
在 Row 32 中，漏掉了“3次尝试失败锁定5分钟”；在 Row 14 漏掉了“偏置行计算”；在 Row 2 漏掉了“文件关联配置”。对于定语过长或补充说明的从句，大模型抓取不全。
3. 修改建议
a. 示例“脱敏”与防捏造指令：把提示词模板里的业务名词（TMDD、MS-DOS）全部替换为与本项目无关的抽象词（如 Http、Linux），并增加最高级警告：“严禁照抄示例词汇”。
b. 加强“顿号/逗号/业务流”切割法：明确指出“登录、校验、分配权限”必须拆，用逗号顿号连接的“锁定、设置、预设”必须拆。
c. 给出“全局/局部”的终极判断标准：提供一个金标准——“这个规则是适用于整个系统的所有功能？还是只适用于某一个页面/组件？”前者进全局，后者留局部。
d. 强化极端细节（数字/异常处理）的抓取：对漏掉重试次数、锁定时间的现象进行专项点名打补丁。

## 修改5
1. 评估结果
平均分：83.50（包含2个满分）
见 evaluation_normal_5.json
2. 问题分析
a. 裁判与干活大模型的“全局/局部”分类标准存在严重分歧
我们在 V10 提示词中教大模型：“如果是针对某个具体功能的性能，放进子需求”。
但裁判大模型认为：
Row 5：显示/滚动 10,000 个节点耗时 ≤5秒，属于【全局性能指标】。
Row 30：ASCII 文件必须包含哪些字段、制表符分隔，属于【全局文件格式要求】。
Row 32：后台配置锁定间隔、Row 39 的数据库高可用和备份，均属于【全局架构要求】。
大模型把它们留在了局部，被裁判无情扣上了“全局污染”或“动词重叠”的帽子。我们需要在提示词里向裁判的分类标准低头。
b. 细微颗粒度不足：“同动词、异宾语”未拆分
大模型对不同动作（增删改）学会拆分了，但对**“相同的动作、不同的操作范围”**依然合并：
Row 32：“浏览当前结果”和“浏览历史结果”被合并。
Row 5：“展开单节点”和“全展开子树”被合并。
大模型认为“反正都是浏览/展开，放一起吧”，但裁判认为这是两个独立动作。
c. 硬核实体与边界值“再次磨损”
报错消息丢失：Row 14 漏了具体的报错词汇 XRT Centroid Error。
时间单位丢失：Row 19 漏了 ≤100微秒。
长列表截断：Row 8 依然把 20 类数据截断成了 17 类；Row 30 漏了 Input Identifier 字段。
3. 修改建议
a. 扩容 global_info 白名单（完全迎合裁判标准）：在提取规则中明确指出，所有的性能指标（耗时、节点数）、所有的文件格式规范（ASCII、分隔符）、所有的系统级备份和高可用机制，一律归入全局信息。
b. 增加“同动词、异宾语”强制切断法则：明确告诉模型：“即使都是浏览，浏览历史和浏览当前必须拆开！即使都是展开，展开单节点和全展开必须拆开！”
c. 加码 AC 的“硬核值校验”：警告模型，如果原文有大写的英文报错（如 XRT Error）、微秒、百分比，必须强制原样抄入 AC。
d. 防长列表截断：进一步强化数量对齐警告。

## 修改6
1. 评估结果
平均分：92.00（包含 5 个满分）
见 evaluation_normal_6.json
2. 问题分析
a. “全局 vs 局部”的范围悖论（裁判标准的震荡）
在上一版（V12）中，为了迎合裁判，我们让大模型把“性能、文件格式、高可用”全塞进全局。
结果在这一版，裁判又说：
Row 30 的“ASCII格式要求”应该放在【导入数据】子需求里。
Row 39 的“24x7高可用”应该放在【处理系统告警】子需求里。
Row 19 的“BIT自检≤10秒”应该放回子需求。
根本原因：裁判判定全局/局部的标准不是看“它是不是性能指标/格式”，而是看“它的作用域（Scope）”。如果一个格式或性能只针对某一个单独的功能，它就必须留在局部！
b. 极值主语与存储介质的“脱落”
大模型现在知道抓取极值了，但经常丢掉极值的“主语”和“地点”：
Row 14 把“计数率<2cps”简写成了“<2cps”（丢了主语）。
Row 19 把“记录在 EEPROM 的 BIT_DATA”简写成了“记录在 BIT_DATA”（丢了存储介质）。
c. “开关动作”触发了裁判的语法误判（动词重叠）
在 Row 19 中，大模型写了“启用并压缩”、“禁用并停止压缩”。裁判看到“启用”和“压缩”两个动词，立刻判定为“动词重叠”。其实大模型想表达的是一个动作（Enable compression），但被中文动宾结构坑了。
3. 修改建议
a. 引入“作用域探测法则（Scope Test）”：彻底重构全局/局部的提取标准。明确告诉大模型：“只要这个限制（哪怕是文件格式、高可用、耗时）只约束特定的某一个操作，就必须留在对应子需求的描述/AC里！只有约束所有操作的，才进 global_info。”
b. 绑定数值与主语/介质：在提取极值的规则中，强制要求连同前面的名词、存储位置一起打包提取。
c. 规范“开关”功能的句式：明确规定“开启/关闭”类动作，必须表述为单一短语（如：“开启数据压缩功能”），严禁使用“启用并压缩”这种复合句。
